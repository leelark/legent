## Current Avoidance Patterns

- Async tests need explicit latches/futures and realistic bounded waits.
- Public auth helpers must stay credentialless/contextless; workspace auth helpers must use explicit context allowlists.
- Tenant/workspace/environment context must be part of every protected backend and frontend workflow.
- High-volume paths must avoid full audience payloads, all-row reads, hot-row contention, and synchronous per-recipient pressure.
- Tracking analytics must distinguish physical raw rows from canonical deduped operational events.
- Kafka high-volume topics need shard-aware keys, local drift tests, and separate production broker evidence.
- Outbox patterns need retention, retry, stuck-row metrics, and target throughput proof.
- Runtime governance decisions must be carried as immutable version/hash snapshots across service handoffs; delivery must fail closed on missing or stale policy evidence.
- Release validators prove local hygiene only; target evidence is still required for promotion.
- Coverage gates should run backend JaCoCo serialized with `-T1`; the repo default `-T1C` can starve local HTTP/reactive unit tests under instrumentation even when scoped tests pass.
- Local HTTP client tests should bind to `127.0.0.1`, use explicit executors, and shut them down; future hardening can replace socket tests with deterministic exchange-function or mock-server stubs.
- Frontend Playwright runs use the standalone production artifact; rebuild with `npm run build:ci` after source changes before trusting E2E snapshots or failures.
- Internal service endpoints discovered from source must have matching public-edge deny rules in both local Nginx and Kubernetes ingress; route-map validation caught the missing audience resolution chunk deny on 2026-05-24 before release claims.
- AI optimization decisions must cross runtime boundaries as campaign-owned, immutable evidence snapshots; the 2026-05-24 frequency slice only applies complete approved evidence as a conservative cap reduction and rejects cap increases for a separate evidence-gated runtime slice.
- Contact/data-extension relationships should be explicit tenant/workspace-scoped metadata, not inferred from matching field names or JSON-only blobs; the 2026-05-24 relationship slice keeps traversal fail-closed until indexed execution and preview/explain contracts are proven.
- Delivery rate-control hot paths should reserve multiple same-scope message tokens in one bounded transaction where possible; the 2026-05-24 batch API preserves warmup/rate safety while reducing per-message lock round trips, but target DB lock/load evidence is still required.
- Segment rule execution should compile to one bounded, explainable plan before count cache lookup or membership mutation; the 2026-05-24 execution-plan slice keeps unsupported relationship traversal fail-closed, rejects missing comparison values and excessive nesting, and does not claim behavioral/event or relationship-backed execution proof.
- Send-time optimization must be an approved immutable campaign snapshot before runtime scheduling can use it; the 2026-05-24 STO scheduler slice fails closed on missing policy/run/hash evidence, low confidence, fallback mode, blocked reasons, timezone/window mismatch, failed launch gates, or scheduled-job mismatch before audience resolution.
- Automation file movement must stay artifact-ID based and storage-adapter mediated; the 2026-05-24 slice rejects raw object references, verifies tenant/workspace object-key prefixes plus size/SHA-256/content type before and after copy, records only compact artifact/checksum metadata, and leaves data-extension extract generation blocked until a provider handoff exists.
- Contact lifecycle audit must be scoped, append-only, and low-amplification: the 2026-05-24 audit slice stores email hashes instead of raw email, scrubs unsafe metadata keys recursively, avoids hard FKs so deletion evidence survives purges, audits explicit eligibility checks as aggregate rows only, and uses aggregate rows for bulk deletes, bulk suppressions, and data-extension overwrite operations.
- Retry/DLQ readiness needs both local operator visibility and target broker evidence: the 2026-05-24 slice adds only low-cardinality `queue`/`source` gauges, validates GA Kafka broker topology evidence, and leaves throughput/replay/alert-delivery claims blocked until target Prometheus/Alertmanager/Grafana and managed Kafka evidence exists.
- AI segment/workflow generation must stay preview-only until apply/runtime paths exist: the 2026-05-24 preview slice uses target-specific RBAC, existing governance/metering audits, hash-only evidence, safe deterministic draft shapes, no provider invocation, no raw output storage, and fail-fast rejection of secret-like context or unsupported relationship/runtime semantics.
- Journey node catalogs can drift ahead of live handlers: the 2026-05-24 `WAIT_UNTIL` slice aligns one advertised advanced node with a runtime handler, validator rules, and editor config while leaving branch/split aliases and side-effect nodes out until their execution semantics and UI authoring are explicit.
- Internal routes that are `permitAll` need more than a shared bearer credential: the 2026-05-24 service identity slice binds the audience chunk read to caller service, tenant, workspace, action, job, chunk, and timestamp before data lookup, while leaving broader internal-route coverage and platform identity as follow-up work.
- Enterprise package movement must not become arbitrary JSON import: the 2026-05-24 package slice limits export to Foundation-owned metadata/configuration object families, requires current tenant/workspace plus source/target environment ownership, blocks unsafe payload indicators before export/validation, verifies manifest/object hashes, and returns dry-run diffs only until live apply governance is separately proven.
- Tracking analytics must separate inspection data from operational metrics: the 2026-05-24 analytics slice keeps exports as physical `raw_events` rows, moves operational counts/timelines/rollups/experiments/goals/reconciliation to canonical `(tenant_id, workspace_id, event_type, id)` semantics, and leaves funnel/segment endpoint semantics plus target runtime evidence as follow-up.
- Segment governance UI must not imply backend enforcement that does not exist: the 2026-05-24 UI slice shows local built-in field classifications, review warnings, locks, and draft-only unsupported families, while persisted segment lock/audit APIs, draft count previews, relationship traversal, event rollups, and recursion enforcement remain follow-up contracts.
